The Token Gate System

Access to YAKK's gated tools is determined by $YST balance at the time of wallet connection. The gate is enforced at two levels: client-side (wallet adapter) and server-side (Helius RPC API route). The server-side gate uses a three-path lookup with fallback and retry to handle RPC rate limiting — a lesson learned from production lockout incidents.

How the Gate Works Technically

The gate operates as a three-path server-side lookup built on Helius RPC:

Path 1 — Server route (/api/balance?wallet=...) performs a getParsedTokenAccountsByOwner call against the Helius private RPC endpoint. 30-second per-wallet in-memory cache. Never 500s — returns DATA_UNAVAILABLE on RPC failure.

Path 2 — Client-side wallet adapter RPC fallback, used only if Path 1 is unavailable.

Path 3 — One retry of the server route after 800ms delay. If all three paths fail, the holder snapshot (lib/holders.ts) is used as a fast-path lookup for known holders. The snapshot is updated from fresh Helius CSV exports and currently covers 17 confirmed wallets.

The rule locked in from production: never trust a single client-side getParsedTokenAccountsByOwner call for gate decisions. Public Solana RPC is not production-grade for token-account reads. Always route balance reads through the private Helius endpoint.

Staking Lock Mechanics

Staking is implemented via an Anchor Rust escrow program. Users lock $YST for defined periods; the contract issues an on-chain rank attestation redeemable by the gate-check API. Longer locks yield higher effective tier scores and accumulate rank points in the Solo Leveling system.

Staking rewards are non-inflationary — sourced from protocol fee revenue rather than mint expansion, preserving $YST's fixed supply economics. Users are never pressured into locking by fear of losing features they already have. Capability-based access only. No manufactured urgency.

For Developers: Environment Variables

The platform requires two environment variables to function in production: NEXT_PUBLIC_RPC_URL (Helius private RPC endpoint) and ANTHROPIC_API_KEY (YAKKAI Coach AI). The Helius key is used across four API routes: /api/balance, /api/gate-check, /api/holders, and /api/stats. The key is never hardcoded. Never exposed client-side. All routes use in-memory TTL caching — never next: { revalidate } in fetch() calls.
